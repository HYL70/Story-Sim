▶
源码证据
# Lines 291-308: 占用角色生成占位意图，但不更新 schedule for cid in active_char_ids: if cid not in intents: # 被跳过 LLM 的角色 occ = self.state.occupancy.get(cid, {}) ... intents[cid] = { "intent": activity, "destination": locked_loc, ... } # ⚠ 这里注释明确写着：占用角色不需要更新 time schedule logger.debug(...)
问题分析

P0-1 Occupancy Model 的实现中，当角色被占用时，代码跳过 LLM 意图生成（line 259 continue），并手动生成占位意图（lines 291-308）。但注释明确写"占用角色不需要更新 time schedule"——这意味着 不调用 _update_character_schedule_from_intent()。

后果链：

  1. 角色 A 在 8:00-9:30 上课（占用中）
  2. 8:30 世界时钟推进，A 被占用 → 使用占位意图，不写 next_event
  3. 9:00 世界时钟再次推进，A 仍被占用 → 又是占位意图
  4. 9:30 之后，busy_until 过期，occupancy 条目被清理
  5. 此时 A 的 personal_schedule 中没有 9:30 之后的事件（因为前两步都没写入）
  6. A 从"活跃角色"列表中消失，被故事遗忘了
  7. 直到 _get_next_world_time() 返回 None → 触发 _initialize_personal_schedules() 回退

严重程度评估：在初始 schedule 还有后续事件的场景下（比如 8:00 上课 + 10:00 课间休息），10:00 的事件仍会触发，所以角色不会完全消失。但如果占用覆盖了 schedule 中最后一个事件（例如 15:00-16:30 社团活动，结束后没有更多 preset 事件），角色确实会失联。发生概率取决于初始 schedule 的覆盖密度。

推荐方案
占用结束时自动写入一个 next_event：
  1. 在占位意图生成处（lines 291-308），仍不调用 _update_character_schedule_from_intent（避免在占用期间写入新事件）
  2. 在 occupancy 清理逻辑处（lines 193-201），当 occupancy 过期被删除时，自动为该角色写入一个"恢复自由"的 next_event：
# 在 occupancy 过期清理后追加： for cid in expired: occ = self.state.occupancy[cid] busy_until = occ.get("busy_until") # 在占用结束时间点之后写入一个自由活动事件 state = self.state.character_states[cid] state.personal_schedule.append(PersonalEvent( day=busy_until[0], hour=busy_until[1], minute=busy_until[2] + 1, duration=60, description="自由活动", importance=3, source="auto", )) state.personal_schedule.sort(...) logger.debug(f"释放 {cid} 占用后写入自由活动事件")

这样确保每个角色在任何时刻都有一个"未来事件"，避免从调度系统消失。

伏笔 buffer 消费顺序为 FIFO — 与叙事上下文无关联
engine/story_engine.py · Line 569-572
新发现 叙事系统 顺序问题
▶
源码证据
# Lines 569-572: 前5条伏笔被消费，不论内容是否与当前场景相关 take = self.state.foreshadow_buffer[:5] foreshadow_texts = [e.get("text", "") ... for e in take] foreshadow_content = "\n".join(foreshadow_texts) self.state.foreshadow_buffer = self.state.foreshadow_buffer[5:]
问题分析

伏笔 buffer 是列表，消费时取 [:5] — 即最老的 5 条先被消费。这意味着：

  1. Step 8 产生的伏笔"走廊上似乎有人跟踪"（与走廊场景相关）
  2. Step 15 产生的伏笔"食堂菜单换了"（与食堂场景相关）
  3. Step 18 叙事场景是【食堂】，但 buffer 先取的是 Step 8 的"走廊跟踪"
  4. 结果：叙事 prompt 被注入了与当前场景语境错位的伏笔，LLM 强行融入会产生违和感

当前的"清理过期 + 消费前5条"机制保证了新鲜度（超过 8 步的会被清理），但没有保证语境匹配度。伏笔与当前场景的关联纯粹靠运气。

推荐方案
方案 A（轻量）：按 created_step 倒序消费
  1. 改为 sorted(buffer, key=lambda e: -e["created_step"])[:5] — 消费最新的 5 条
  2. 好处：最新伏笔更可能与当前叙事语境相关
  3. 改动极小（1 行）
方案 B（更精确）：按场景关键词匹配
  1. 对当前 active_interactions 提取关键词（角色名 + 位置名）
  2. 对每条伏笔的 text 计算关键词命中数
  3. 优先消费命中数最高的伏笔，其次按 recency
  4. 缺点：增加计算量，且关键词可能误导匹配
推荐方案 A — 改动成本最低，逻辑上合理的假设是"越新的伏笔越有可能与当前上下文相关"。

N5
_pop_scheduled_events 的"每个角色只取一个事件"保护遗留重复事件
engine/story_engine.py · Line 659-675
新发现 调度系统 边界情况
▶
源码证据
def _pop_scheduled_events(self, day, hour, minute=0): events = {} for cid, state in self.state.character_states.items(): remaining = [] for evt in state.personal_schedule: if evt.day == day and evt.hour == hour and evt.minute == minute \ and cid not in events: # ← 只取第一个！ events[cid] = {...} else: remaining.append(evt) # 第二个同时间事件被保留 state.personal_schedule = remaining
问题分析

cid not in events 的保护确保每个角色在同一分钟只取一个事件。但如果由于 bug 或 LLM 输出异常导致同一角色在同一分钟有两个事件，第二个被保留在 personal_schedule 中。

后果：

  1. 当前步：第一个事件被消费，第二个仍留在 schedule[0]（同一天同一分钟）
  2. 下一步：_get_next_world_time() 再次找到同一个 (day, hour, minute)
  3. 世界时钟"原地踏步"——在同一天同一分钟内再次触发
  4. 如果 LLM 再次为这个角色生成同时间 next_event，可能形成无限循环

触发概率：低——需要在同一分钟出现重复事件。这通常发生在：初始化时 LLM 返回了重叠时间的 schedule、或游戏加载旧存档数据异常。但在大量运行中偶尔会发生。

推荐方案
在 pop 时同时清理所有同时间的事件（而非只取第一个），并记录 warning： def _pop_scheduled_events(self, day, hour, minute=0): events = {} for cid, state in self.state.character_states.items(): remaining = [] taken = False for evt in state.personal_schedule: if evt.day == day and evt.hour == hour and evt.minute == minute: if not taken: events[cid] = {...} taken = True else: logger.warning(f"重复事件: {cid} 在 d{day} {hour}:{minute} 有多个事件，丢弃") # 不加入 remaining，直接丢弃 else: remaining.append(evt) state.personal_schedule = remaining

同时，在 _update_character_schedule_from_intent 和 _initialize_personal_schedules 中加入同时间去重（写入前检查是否已有同时间的 event）。

N6
叙事 prompt 仅含 system 角色 — DeepSeek 对 system message 的 JSON 遵守率较低
prompts/narrative.py · Line 158
新发现 LLM 交互 prompt 工程
▶
源码证据
# prompts/narrative.py line 158 — 唯一返回的 message: return [{"role": "system", "content": system_prompt}] # 对比：character_action.py line 127-131 — 同样只有 system: messages = [ {"role": "system", "content": system_prompt}, ]
问题分析

DeepSeek API 文档中，system 角色定义为系统级指令，user 角色定义为用户查询。当整个 prompt 都在 system message 中、没有 user message 时，模型有时会"忘记"它应该按照 JSON 格式输出。这在实践中表现为：

  • 输出纯文本叙事而不包装在 JSON 的 narrative 字段中
  • 忘记输出 foreshadowing 字段
  • 在 JSON 之前添加解释性文字（"好的，这是叙事文本："）

llm_client.parse_json_response() 有兜底解析（正则提取），但这些兜底只能恢复 narrative 文本，foreshadowing 字段丢失。

对比：character_action.py 和 reflection.py 的 prompt 也是仅 system role。所有 LLM 调用点都面临同样的风险。

推荐方案
在 system 之后追加一个简短的 user message 作为"触发器"： return [ {"role": "system", "content": system_prompt}, {"role": "user", "content": "请按照要求输出 JSON。"} ]

同样修改 character_action.py、reflection.py、world_sim.py、auto_event.py、world_event_judge.py 中所有仅含 system message 的 prompt。

N7
_initialize_personal_schedules 跨日判断过于简单 — LLM 输出绝对小时被误判
engine/story_engine.py · Line 728-732
新发现 时间系统 边界情况
▶
源码证据
# Lines 728-732: h = int(item.get("hour", base_hour + 2)) % 24 ... actual_hour = h actual_day = self.state.current_day if actual_hour <= base_hour: # ← 唯一判断标准 actual_day += 1
问题分析

跨日判断仅依赖 actual_hour <= base_hour。这在简单场景下正确，但有两个边界问题：

  1. 多日跨度：如果 LLM 返回 hour=8 而 base_hour=20，合理推断是"明天 8 点"（+1 day）。但如果其实是"后天 8 点"（因为 schedule 应该覆盖 2 天），系统无法区分。虽然 prompt 中说"today or next day"，但 LLM 可能不严格遵守。
  2. 同日但小时早于 base_hour：如果 base_hour=12，LLM 返回 hour=10，系统判定为 cross-day（+1 day）。但如果是在同一天的上午 10:00 回顾？这在正常流程中不会发生（世界时间单调递增），但从存档恢复时可能出现。

实际上，schedule 初始化的 prompt 中有 must be > current hour {base_hour} today，所以正常情况下 LLM 不会返回小于 base_hour 的小时。但 LLM 不总是遵守约束。

推荐方案
增强跨日判断逻辑： def _resolve_schedule_day(self, raw_hour, base_hour, base_minute, current_day): # 计算目标分钟数 vs 当前分钟数 target_total = raw_hour * 60 current_total = base_hour * 60 + base_minute if target_total <= current_total: # 目标时间在今天已经过去 → 安排到明天 return current_day + 1, raw_hour else: # 在今天未来 return current_day, raw_hour

将分钟也纳入比较，消除边界歧义。同时记录 warning 当检测到跨日事件以便调试。

N8
无 LLM 并发控制 — 8+ 角色并行调用可能触发 API 限流
story_engine.py:269 + Step 3 对话 + Step R 反思
新发现 并发 API 限流
▶
问题分析

每步的 LLM 调用量分析（8 角色场景）：

  • Step 2 角色意图：最多 8 个并行调用（asyncio.gather）
  • Step 3 对话生成：取决于碰撞数，通常是 1-4 个并行
  • Step R 反思：所有 active 角色同时反思（每 3 叙事步）→ 最多 8 个
  • 世界事件：auto_event + impact_judge = 2 个调用（每 12 步）

在最坏情况：8 个意图 + 4 个对话 + 8 个反思 + 2 个事件 = 同一时间步中 22 个并发 LLM 调用。DeepSeek API 没有公开的硬并发限制，但在高并发场景下可能出现：

  1. 429 Too Many Requests — 单次调用失败，虽然有 retry（指数退避），但 22 个并发重试会加剧拥堵
  2. 响应时间显著增加 — 服务器端排队
  3. Token 消耗集中爆发 — 不利于成本控制
推荐方案
引入 asyncio.Semaphore 限制并发数： # 在 StoryEngine.__init__ 中: self._llm_semaphore = asyncio.Semaphore(6) # 最多 6 个并发 # 在每个 LLM 调用点包裹: async with self._llm_semaphore: response = await llm_client.chat_async(...)

这也解决了 N4 中提到的问题：如果部分调用失败/超时，不会因为 22 个重试同时爆发而导致级联失败。

问题分析

当玩家传入 hour 参数时（旧 API 路径），事件被同时写入所有角色的 personal_schedule。这意味着玩家设定一个"下雨"事件在 14:00 发生，所有 8 个角色的 schedule 中都会出现这个事件。

这与 per-agent schedule 的核心理念冲突：每个角色应该有自己的时间安排，"下雨"是世界事件而非角色事件。写入所有角色的 schedule 会导致：

  1. 所有角色在同一时间被标记为"活跃"
  2. LLM 为所有角色生成意图时，scheduled_event 都是"下雨"
  3. 角色原本的个人安排被覆盖

正确的做法是：玩家事件应该通过 pending_player_events → process_pending_events → judge_event_impact 路径处理，由系统判定哪些角色受影响。

推荐方案
统一两条路径，废弃旧路径：
  1. 将 hour 参数从 add_player_event 中移除或标记为 deprecated
  2. 所有玩家事件统一走 world_event.add_player_event → pending_player_events 路径
  3. 如果玩家确实需要指定时间，在 WorldEvent 中增加 scheduled_hour 字段，在 process_pending_events 中检查时间是否到达才激活

如果旧 API 路径仍有使用者（例如 Gradio UI 中的手动触发），先确认后再决定是修改 UI 还是保留兼容。
