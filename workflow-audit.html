🔴 P0 — 架构级问题（影响系统正确性）

P0
Duration 被忽略 — 世界时钟不跟踪"进行中"的活动
Step 1 · _get_next_world_time() · engine/story_engine.py
时间系统状态一致性
▶
问题分析
这是最严重的架构缺陷。每个角色的 PersonalEvent 有 duration 字段（表示活动持续分钟数），但世界时钟完全不使用它。 具体场景：
  1. 角色 A 在 8:00 开始上课（duration=90min，到 9:30）
  2. 角色 B 在 8:30 下课，世界时钟跳到 8:30
  3. 此时 Step 2 会为所有角色生成意图——包括正在上课的角色 A
  4. 角色 A 可能在 8:30 决定"去小卖部"——这完全破坏了合理性
根因：系统用"事件开始时间"驱动世界时钟，但完全忽略了"事件进行中"的状态。角色的 current_location 没有与 duration 锁定，LLM 可以在活动进行中生成完全背离的意图。
推荐方案
方案：引入"占用状态"（Occupancy Model） 在 StoryState 中维护 occupancy: dict[str, dict]： occupancy = { "character_a": { "busy_until": "08:00-09:30", # 基于 total_minutes 计算 "locked_location": "教室A", "activity": "数学课" } } 在 Step 1 _get_next_world_time() 之后、Step 2 之前：
  1. 根据当前时间清理过期的 occupancy 条目
  2. 对活跃角色的事件，如果有 duration，写入新 occupancy 条目
  3. Step 2 中，被占用的角色 不调用 LLM，直接使用 locked_location 和 activity 作为 intent
  4. 被占用角色的 importance 设为当前活动的 importance（而非默认 2）
这与 Inworld AI 的 character state machine 思路一致——角色在"进行中"的状态下不应自主决策。
P0
时间倒走风险 — next_event 语义歧义 + 无倒退保护
Step 2 → Node S · _update_character_schedule_from_intent()
时间系统数据校验
▶
问题分析
语义混乱：next_event_hour 在不同地方的语义不一致：
  • 在 _initialize_personal_schedules() 中：hour 是绝对小时（0-23）
  • 在 character_action.py prompt 中："几小时后（自行判断）"——这是 相对偏移
  • 在 _update_character_schedule_from_intent() 中：total_minutes = current_hour*60 + offset_hours*60——当做相对偏移处理
两种不一致同时存在：
  1. prompt 中"几小时后"与初始化 prompt 中"0-23"冲突——LLM 可能输出绝对小时或相对小时，无法区分
  2. 即使正确当做相对偏移，如果 LLM 返回 next_event_hour=0（"0小时后"），事件被安排在同一分钟，导致角色事件堆积
  3. 如果 LLM 错误地将绝对小时 22 当做相对偏移计算：当前 20:00，偏移 22 → total=42h → 第二天 18:00，产生巨大跳跃
更隐蔽的问题：没有时间防倒退校验。如果 LLM 输出的 next_event 计算出的时间点早于当前世界时间，会被写入 personal_schedule，导致世界时钟在下一步跳回过去。
推荐方案
方案 A：统一时间语义（优先）
  1. 将 next_event 全部统一为绝对时间表示：day, hour, minute
  2. prompt 改为："下一个安排在哪天几点几分（day/hour/minute，必须晚于当前时间 {current_day}天{current_hour}时{current_minute}分）"
  3. 解析时直接用绝对时间构造 PersonalEvent，不再做偏移计算
方案 B：时间防倒退校验（兜底）
  1. _update_character_schedule_from_intent() 末尾，解析出的 new_day/hour/minute 与当前世界时间比较
  2. 如果计算出的事件时间 ≤ 当前时间，自动增加 duration 分钟作为最小偏移（如 +10min）
  3. 记录 warning 日志
方案 C：next_event 最小间隔
  1. 如果 LLM 返回 next_event_hour=0 且 next_event_minute=0，强制最小值 = 5 分钟
  2. 防止角色事件在同一分钟内无限堆积
P0
LLM 调用失败无优雅降级 — 单点故障导致整步中断
Step 2/3/5/E/R · 所有 LLM 调用点
错误处理稳定性
▶
问题分析
当前代码中，所有 LLM 调用如果失败（网络超时、Rate Limit、JSON 解析失败），异常会向上传播到 advance_one_step()，导致整步失败。具体影响：
  • Step 2 单角色失败 → N 个角色全部损失：虽然并行调用是独立的，但如果 asyncio.gather 中一个任务抛异常且 return_exceptions=False，所有结果丢失
  • 叙事生成失败 → 该步完全没有输出：即使前面的交互、碰撞、对话都已正确计算
  • 没有 retry 机制：DeepSeek API 偶尔返回 malformed JSON 是已知现象，但代码中没有针对性的 retry
发生概率：以 8 个角色、每步 5-8 次 LLM 调用计算，运行 50 步 = 250-400 次调用。即使单次成功率 99%，累积失败概率也达到 92-98%。
推荐方案
分层防护策略： 1. Step 2 — 单角色隔离
  1. 确保 asyncio.gather 使用 return_exceptions=True
  2. 失败的角色的 intent 使用占位意图：intent="继续当前活动", destination=current_location, importance=2
  3. 失败的角色的 next_event 设为 current_time + 30min（让它在下次时钟推进时重新尝试）
  4. 记录失败日志，不阻塞其他角色
2. 叙事生成 — fallback 叙事
  1. LLM 调用最多 retry 2 次（每次 3s 间隔）
  2. 3 次都失败 → 生成模板叙事："{时间}。{角色名}在{位置}{事件描述}。"
  3. 模板叙事同样写入 timeline，标记 source="fallback"
3. JSON 解析 — 结构化 retry
  1. LLM 返回后先尝试 json.loads()
  2. 失败 → retry 一次（在 prompt 末尾加 "OUTPUT VALID JSON ONLY"）
  3. 仍然失败 → 使用 ast.literal_eval() 或正则提取关键字段
P0
目的地自由文本 — 碰撞检测按字符串匹配，精度极低
Step 3 · detect_encounters() · engine/world_sim.py
碰撞检测字符串匹配
▶
问题分析
碰撞检测按 destination 字段分组，但 destination 是 LLM 自由生成的文本： "图书馆" vs "学校图书馆" → 不匹配 "图书馆" vs "図書館" → 不匹配 "走廊" vs "走廊上" → 不匹配 这会导致大量假阴性（本应相遇的角色被判定为擦肩而过），以及更隐蔽的假阳性（LLM 用词巧合导致两个完全不同地点被匹配，如"楼顶"和"实验楼顶层"）。 实际影响：在 8 角色场景中，如果每个角色独自在不同措辞中引用同位置，碰撞发生率可能从预期的 30-40% 降到 5-10%，严重影响角色互动。
推荐方案
方案 A：地点规范化（推荐，改动最小）
  1. 在 StorySetting 中定义 标准化地点列表（校园类约 10-15 个标准地点）
  2. 在 character_action.py prompt 中，将 destination 输出格式约束为从预设列表中选择： "destination": "从以下地点中选择一个：教室, 走廊, 图书馆, 食堂, 操场, 体育馆, 社团活动室, 屋顶, 校门口, 教师办公室, 保健室, 中庭"
  3. LLM 仍可自由选择，但词汇表被限制，保证匹配精度
  4. 提供 1-2 个通用 fallback："其他室内"、"其他户外"
方案 B：语义匹配（更灵活，更复杂）
  1. 使用简单的嵌入向量（sentence-transformers）计算 destination 之间的余弦相似度
  2. 相似度 > 0.85 视为同地点
  3. 缺点：引入额外依赖，运行时开销
推荐方案 A — 校园剧场景地点有限，预设列表是最简单有效的方案。

🟠 P1 — 设计缺陷（影响输出质量和体验）

P1
叙事按场景独立生成 → 时间线顺序被破坏
Step 5 · _generate_narrative_with_foreshadowing()
叙事质量时序
▶
问题分析
叙事按 location 分组后独立生成。如果场景 A 的事件发生在场景 B 之前，但叙事顺序是"图书馆"→"教室"，读者会先读到逻辑上后发生的事。 具体场景：
  1. 10:00 — 角色 A 从走廊去图书馆，路上与角色 B 发生碰撞
  2. 10:00 — 角色 C 在教室自习，角色 D 进来……
  3. 叙事输出：先写"【教室】……"，再写"【图书馆】……"
  4. 读者困惑：角色 A 不是在走廊遇到 B 了吗，怎么已经在教室了？
虽然它们都是 10:00 的事件（"同时"），但在读者感知中，"走廊偶遇"应该是发生在"进入图书馆"之前的。分组独立生成完全丢失了这个因果链。
推荐方案
方案：统一单次 LLM 调用生成所有场景叙事
  1. 不再按场景独立调用 LLM，而是将所有场景的互动数据一次性传入同一个 narrative prompt
  2. prompt 中按角色时间线排序（而不是按位置排序），并提供时间顺序指导： 【本时间段的事件时间线】 10:00 — {角色A}在走廊遇到{角色B}，简短对话后前往图书馆 10:00 — {角色C}在教室自习，{角色D}进入 请按时间线顺序写作，先发生的先写。
  3. 输出仍保持场景分组格式：{"narrative": "【走廊】……\n\n【教室】……\n\n【图书馆】……"}
  4. 这样 LLM 有全局视角，可以在场景间做自然过渡
权衡：单次调用 token 量更大（所有场景数据一起传入），但换来了时序正确性和场景间过渡质量。如果场景过多（>4个），可以分批 2-3 个场景一组，但组内仍保持统一调用。
P1
角色对话生成无全局一致性 — 同一角色的多个对话可能矛盾
Step 3B · generate_dialogue() · engine/world_sim.py
对话质量一致性
▶
问题分析
对话生成是按碰撞对独立并行的。如果角色 A 在同一时间步与角色 B 和角色 C 都有 DIALOGUE 碰撞：
  1. LLM 调用 A-B 对话：A 说"我讨厌下雨天"
  2. LLM 调用 A-C 对话：A 说"我最喜欢下雨天了"
两个对话是独立 prompt，互不感知。A 在两个对话中表现出完全矛盾的态度，读者会察觉。 更隐蔽的问题：A-B 对话的 outcome 可能涉及情绪变化（"A 变得沮丧"），但 A-C 对话使用的是 A 调用前的情绪，不会反映这个变化。
推荐方案
方案：对话前注入角色的"当前状态摘要" + 情绪共享
  1. 在所有对话 LLM 调用之前，先生成一个"角色初始状态快照"：每个角色的当前心情、最近记忆、意图摘要
  2. 将"状态快照"注入每个对话 prompt，确保所有对话都从同一基线出发
  3. 对话结果收集后，如果同一角色参与多段对话，按顺序（先发生的）更新其情绪描述，后续对话以更新后的情绪为输入
  4. 如果对话数量少（≤3），可以尝试合并为一次 LLM 调用：将同一位置的所有碰撞合并为一个"集体对话" prompt
简单版（最小改动）：在对话 prompt 中增加约束： "{角色A}可能正在或即将与其他人互动。请确保其在本段对话中的态度、 心情和言论与角色设定一致，不与其在当前时间步的其他互动冲突。"
P1
记忆淘汰逻辑缺陷 — 二元 is_reflection 排序破坏重要性优先级
Step 4 · memory.add_memory() · engine/memory.py
记忆系统算法缺陷
▶
问题分析
记忆淘汰排序键：(is_reflection, importance, step) 降序。 问题：is_reflection 是布尔值（True/False）。排序后，所有反思记忆（不论 importance 多低）都排在所有普通记忆（不论 importance 多高）之前。 后果：一个 importance=1 的冗余反思记忆不会被淘汰，而一个 importance=9 的关键事件记忆却被淘汰。 案例： 记忆 A: is_reflection=True, importance=1, step=50 → 保留 记忆 B: is_reflection=True, importance=2, step=49 → 保留 记忆 C: is_reflection=False, importance=9, step=48 → 淘汰 ❌ 记忆 D: is_reflection=False, importance=7, step=47 → 淘汰 ❌
推荐方案
方案：加权排序替代布尔优先 将 is_reflection 从排序键改为权重加分： def add_memory(self, memories, new_memory, max_count=100): if len(memories) >= max_count: # 加权评分替代二元排序 def score(m): base = m.get("importance", 0) * 10 # 重要性权重 x10 base += m.get("step", 0) # 近期性 if m.get("is_reflection"): base += REFLECTION_BONUS # 反思固定加分 (e.g., 30) return base memories.sort(key=score, reverse=True) memories = memories[:max_count] memories.append(new_memory) 这样反思记忆有优势但不绝对——一个 importance=9 的普通记忆（得分 90）会高于 importance=1 的反思记忆（得分 10+30=40）。 REFECTION_BONUS 建议设为 20-30（相当于普通记忆 2-3 点 importance 的提升）。
P1
伏笔缓冲区无过期机制 — 陈旧伏笔永远不会被清理
Node A · foreshadow_buffer · Step 5 消费
叙事系统内存管理
▶
问题分析
伏笔 buffer 的流程：
  1. Type B (foreshadow) 模式：LLM 生成伏笔 → 写入 buffer
  2. 正常叙事：取出最多 5 条伏笔注入 prompt → 但只有被"消费"的才清空
问题：如果连续多步都是 cooldown/full 模式，步中 importance 一直 ≥ 4，那伏笔 mode 永远不会触发。已生成的伏笔堆积在 buffer 中，没有过期机制。 更隐蔽的问题：伏笔可能已经"语境过时"。例如第 10 步生成的伏笔"走廊上似乎有人跟踪"——到了第 25 步，故事已经发展到了午休时间，这个伏笔的语境已经不存在了，但系统仍会在下一次叙事中尝试融入。
推荐方案
方案 A：伏笔过期机制
  1. 每条伏笔追加 created_step 字段
  2. 在每次叙事前，清理超过 FORESHADOW_MAX_AGE（建议 8 步）的伏笔
  3. 过期的伏笔写入角色记忆（importance=3，作为"有过这个想法但没发生"的标记）
方案 B：伏笔强制消费窗口
  1. 如果 buffer 非空且连续 3 步未触发叙事（skip），在第 4 步强制触发一次 light 叙事（即使 importance < 4）
  2. 在 light 叙事中注入伏笔
  3. 这样伏笔不会无限期等待，有一个强制消费窗口
推荐方案 A + B 结合：过期清理 + 强制窗口，双重保险。
P1
角色并行决策互不感知 — 可能导致逻辑冲突
Step 2 · _generate_character_intent() 并行调用
一致性并行
▶
问题分析
角色意图生成是完全并行的——每个角色看到的"世界状态快照"都是一样的（同一步快照），但不知道其他角色在同一时间步决定做什么。 冲突场景：
  1. 角色 A 决定"去图书馆找 B 道歉"
  2. 角色 B 同时决定"去校门口等 C"
  3. A 到了图书馆 → B 不在 → A 的意图失败但系统不知道
当前 Step 3 的碰撞检测只能解决"同位置"的碰撞，不能解决"意图指向某角色但目标不在"的情况。intent 中的"找 B"意图被忽略了。
推荐方案
方案：意图后处理 — 交叉校验 在 Step 3 碰撞检测之前，加一个轻量规则层：
  1. 解析所有角色的 intent，识别"目标指向型意图"：正则匹配 "找/见/等/约 + 角色名"
  2. 如果角色 A 的意图指向角色 B，但角色 B 的 destination ≠ 角色 A 的 destination → 意图失败
  3. 意图失败的角色的 actual_action 追加 "（未遇到 {B}）"，写入记忆（importance=3）
  4. 可选：触发角色 A 的备选行动（LLM 轻量 re-roll）或直接标记为"等待/空转"
这是一个纯规则处理，不增加 LLM 调用，实现成本低。

🟡 P2 — 健壮性与边界问题

P2
Token 膨胀 — 上下文随步数线性增长
Step 2 · 角色意图 prompt + Step 5 · story_summary
性能token 成本
▶
问题分析
Step 2 每个角色的 prompt 包含：
  • 完整角色档案：~500 tokens
  • 8 条记忆：~800 tokens
  • 关系摘要（7 个角色）：~500 tokens
  • 世界快照（所有角色位置/心情）：~600 tokens
  • Scratchpad：~300 tokens
  • World events：~200 tokens
  • System prompt：~300 tokens
  • 合计：~3200 tokens/角色
Step 5 叙事 prompt：
  • story_summary（50步 × 200字）：~6000 tokens
  • 角色卡：~800 tokens
  • 互动数据：~500 tokens
  • System prompt：~400 tokens
  • 合计：~7700 tokens
8 角色 × 3200 = 25600 tokens（Step 2）+ 7700（Step 5）= ~33k tokens/步（不含对话生成和反思）。 DeepSeek-chat 上下文上限 64k tokens。50 步后，仅 story_summary 就占 ~6k tokens，加上 prompt 基础部分（~10k），目前仍有余量。但 memory 的检索结果随故事推进越来越长，最终会超出限制。
推荐方案
短期（无需改架构）：
  1. 记忆检索从 8 条降到 5 条（最新 3 条 + 最重要 2 条）
  2. story_summary 截断从 200 字/条降到 100 字/条，或从 50 步降到 30 步
  3. 关系摘要只包含"有变化的关系"，无变化的不列出
中期（需改架构）：
  1. 引入 LLM 压缩：每 20 步调用一次摘要 LLM，将 story_summary 从 50×200字 压缩为一段 600-800 字的精炼摘要
  2. memory 分"热记忆"（最近 5 步，完整）+ "冷记忆"（摘要化，只保留关键信息）
长期参考 — Stanford 论文做法： 使用 embedding 向量做记忆检索，只取最相关的 3 条而非固定 8 条，大幅减少 token 消耗。
P2
cooldown 状态无 re-entry 保护 — 连续高压事件导致状态混乱
Node A · arc 状态机 · engine/story_engine.py
状态机边界情况
▶
问题分析
当前状态机： normal → [imp≥7] → cooldown (3步) → [3步后] → check → normal 问题场景：
  1. Step 5: imp=8 → 进入 cooldown，cooldown_remaining=3
  2. Step 6: (cooldown) imp=7 → cooldown_remaining=2 ✓（正常递减）
  3. Step 7: (cooldown) imp=9 → cooldown_remaining=1 ✓
  4. Step 8: (cooldown) cooldown_remaining=0 → 进入 check
这个看起来正常。但：
  1. 如果在 cooldown 期间（Step 6），又出现了一个 need_recalc 标记（比如世界事件触发）
  2. 系统可能重新计算弧状态 → 可能把 cooldown_remaining 重置回 3
  3. 结果：cooldown 无限延长，故事永远无法回到 normal
根因：状态机没有"re-entry 保护"——进入 cooldown 后，除非超过 cooldown_remaining，否则不应该允许任何条件重新触发 cooldown 入口。
推荐方案
方案：cooldown 期间忽略高重要性重入 def _update_arc_state(self, max_importance, has_events, has_encounters): state = self.state if state.arc_state == "cooldown": state.cooldown_remaining -= 1 # 🔒 cooldown 期间不响应任何状态切换 if state.cooldown_remaining <= 0: state.arc_state = "check" return # 直接返回，不继续判断 if state.arc_state == "normal": if max_importance >= ARC_HIGH_IMPORTANCE: state.arc_state = "cooldown" state.cooldown_remaining = ARC_COOLDOWN_STEPS # 记录进入 cooldown 时的最大 importance state.arc_triggered_by = max_importance return 关键改动：
  1. cooldown 状态的处理放在最前面，return 后不再继续判断
  2. 不再允许 normal → cooldown 的重入（即使 imp≥7，在 cooldown 期间也忽略）

反思触发条件：current_step % 3 == 0 且 有叙事。 场景：
  1. Step 9-10-11: 全是 skip（importance<4，无碰撞）→ 无叙事 → 反思不触发

auto_event 无消费保障 — 可能生成后立即过期

问题分析
auto_event 生成周期：steps_since_auto_event >= 12 事件有效周期：active_until_step = current_step + 5 (plot) / +3 (env) 问题：如果生成的事件没有影响任何角色（_judge_event_impact() 返回空 affected_characters），事件就变成了"幽灵事件"——它存在但被忽略了，然后过期消失。 更糟的情况：事件影响了角色，但因为接下来的几步恰好都是 skip（importance<4），事件在 cooldown 期内过期，没有被写入任何叙事。 浪费：每次 auto_event 消耗 2 次 LLM 调用（生成 + 影响判定）。如果 30% 的事件没有被消费，相当于浪费了 30% 的事件 LLM 成本。
推荐方案
方案 A：事件触发条件叙事
  1. 当 auto_event 有 affected_characters 时，将该步的 narrative_mode 强制设为 "full"（覆盖 arc 状态）
  2. 这样确保事件被叙事消费，不会无声过期
方案 B：事件持续性追踪
  1. 事件被 affected 角色写入其记忆（importance=event importance），即使没有叙事
  2. 在下次叙事时，LLM 可以通过 story_summary 中的记忆引用了解事件
  3. active_until_step 延长到 8 步（给 skip 序列更多时间窗口）
推荐 A——最直接有效。

📋 综合改进建议

1. 引入"World Time Controller"抽象层
将时间推进逻辑从 engine 中独立为 engine/world_clock.py，统一管理 occupancy、时间防倒退、跨日计算。当前这些逻辑散布在 4-5 个方法中，耦合度高。
2. 建立 Location Registry
在 config.py 中维护标准化地点列表，所有 prompt 的 destination 字段强制从列表中选择。同时为每个地点维护"可同时容纳人数"，用于碰撞检测优化。
3. 统一 LLM 调用的 retry + fallback 层
封装一个 llm_call_with_retry(prompt, max_retries=2, fallback=None) 函数，所有 LLM 调用点统一使用。内置 JSON 解析重试和模板 fallback。
4. 每 10 步自动保存 checkpoint
当前存档全靠手动。如果 LLM 调用在 Step 7 失败，Step 1-6 的计算结果全部丢失。每 10 步自动保存一个 .checkpoint 文件，失败后可以从最近的 checkpoint 恢复。
5. 增加调试面板的"步内状态"视图
在 Gradio 界面中增加一个可折叠面板，显示当前步的：所有角色的 intent、碰撞结果、弧状态、narrative_mode 选择理由。这对调试时间问题和碰撞问题至关重要。
