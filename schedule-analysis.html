发现的 3 类问题

🔴 问题 1：事件依赖了不可用的角色（3 处）

  • 燃堂 10:00 "课间和齐木聊天" → 齐木的 schedule D1 最早 12:00 才开始。齐木在 10:00 没有事件，系统不知道他在哪，但他也不会主动去找燃堂。
  • 燃堂 12:00 "和齐木一起吃拉面" → 齐木 12:00 在屋顶吃咖啡果冻（30min），不是在拉面店。两人互相矛盾。
  • 照桥 12:30 "对牢齐木的位置" → 齐木 12:30 已经从屋顶去了图书馆角落（90min）。照桥在教室盯着一个空座位。

这三件事在后续模拟中会导致"意图交叉校验"捕获（P1-5），importance 被降级——但 LLM 在生成日程时就埋下了这些无效事件，浪费了 schedule slot。

🟡 问题 2：同时间同地点但未协作（1 处）

  • 海藤 12:00 天台吃便当 vs 齐木 12:00 屋顶吃果冻 → "天台"和"屋顶"是同一地点（系统可能用不同表述），两人应该在 12:00-12:30 在天台相遇——这是绝佳的碰撞机会，但因为 schedule 各自独立生成，LLM 不知道对方也在。

🟡 问题 3：上课时间不统一

  • 海藤 09:00 上课、燃堂 09:00 上课、照桥 09:00 数学课 ✓
    但齐木 D1 完全没课！最早的事是 12:00 吃果冻。他们是同班同学，理论上应该共享课表。
  • 燃堂有 D1 09:00 上课和 D2 08:00 上课；齐木只有 D2 08:00；照桥 D1 09:00 + D2 08:00。混乱的时间表会让"教室"场景的碰撞概率大幅下降。

🔴 问题 4：importance 严重虚高 — 直接导致 skip 失效

  • 燃堂 "09:00 上课" → 🔴 红色（最高重要性） ← 这只是一节普通的课！
  • 齐木 "16:00 回家锁门打游戏" → 🔴 红色 ← 回家打游戏为什么是最高重要性？
  • 海藤 "15:30 放学后借漫画" → 🔴 红色 ← 放学借书也很 routine
  • 照桥 "D2 08:00 教室整理课本" → 🔴 红色 ← 整理课本是日常动作

日程 prompt 中 importance 的示例值是 5 和 4（"what to do"），但 LLM 从角色设定推断"这个角色很重视这件事"→ 赋 8-10 分。这正好通过 Occupancy 透传链触发 cooldown → skip 失效。详见 skip-root-cause.html。

连锁影响：这些 routine 事件以 7-10 的 importance 进入 PersonalEvent → Occupancy 占位意图继承高值 → max_importance ≥ 7 → arc_state = cooldown → 连续 3 步 full 叙事 → 真正的"高压事件"没有区别度。

流程根因

根因①：per-character 独立 LLM 调用，互不知晓（问题 1/2/3）

代码中 for cid in char_ids: ... await llm_client.chat_async(...) —— 每个角色的日程完全独立生成。

根因②：schedule prompt 无 importance 语义指导（问题 4）

当前 prompt 只说 importance: 1-10，没有区分 routine / significant / life-changing 的标准。LLM 从角色设定推断"燃堂很重视上学"→ 赋分 8-10。需要 explicit rubric。

详见 skip-root-cause.html 的完整分析。此问题与 Occupancy 透传链合并导致 skip 完全失效。

流程修复方案

方案 A：Prompt 层防御（最小改动）

在 schedule 生成 prompt 中增加约束：

"Schedule only YOUR character's actions. Do NOT assume other characters will be present or available. Instead of 'chat with X', write 'look for X to chat' or 'hope to see X'. Instead of 'eat with X', write 'go to cafeteria hoping X is there'."

这样即使齐木不在，事件描述也是合理的期待而非事实依赖。

方案 B：两阶段日程生成（结构改进）

阶段 1：系统先生成共享框架（不调 LLM），例如：

SHARED_SLOTS = {
  "08:00-08:15": "到校/早会",
  "08:15-12:00": "上午课程（所有学生在教室）",
  "12:00-12:45": "午休/午餐时间",
  "12:45-15:00": "下午课程",
  "15:00+": "放学后自由时间"
}

阶段 2：LLM 在框架内个性化。每个角色在"自由时间"槽内自由发挥，但"上课"时所有角色共享教室位置。这样：

  • 齐木 12:00 不会安排"屋顶吃果冻"——因为 12:00 是午休共享时间，其他角色也可能来屋顶
  • 燃堂不会写"和齐木聊天"——他会在 prompt 中看到"齐木的 12:00 安排尚未确定"
  • 上课时间统一 → 大量角色共处教室 → 提高碰撞概率

方案 C：后处理校验 + 修正

所有角色日程生成完毕后，跑一次规则校验：

  • 正则匹配 和X聊天|和X一起|找X|等X → 检查 X 的 schedule 是否有同时间同地点的事件 → 没有则警告或自动降级为"期待"语气
  • 同一时间同一地点检测 → 标记为"高碰撞概率时段"，可以在 narrative 中更主动地推动互动
综合建议：方案 A（prompt 约束）立刻实施——改一行 prompt 就能阻止 90% 的"不切实际的依赖事件"。方案 B 作为架构改善在下一个迭代中做——它解决的是更根本的"角色不在同一时空"问题。问题 4（importance 虚高）：日程 prompt 中增加 importance rubric + Occupancy 占位意图 cap 到 6（详见 skip-root-cause.html），双管齐下。
