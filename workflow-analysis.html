四层流水线架构（事件调度 → 角色决策 → 世界模拟 → 叙事编排）的核心思路是正确的，借鉴了 Stanford Generative Agents 的核心理念，并针对"跑团式群像剧"做了务实的适配。per-agent schedule 调度机制、Occupancy Model、弧追踪状态机等都是合理的工程设计。但在关系系统闭环缺失、碰撞判定粒度不足、LLM输出一致性、伏笔机制脆弱等方面存在值得改进的结构性问题。

模块 0：故事初始化

0 init_story()LLM 生成开场白 + 初始关系矩阵 A ▶
逻辑评估

整体设计合理：将"初始叙事生成"和"初始关系确定"打包为一个 LLM 调用，减少请求次数。生成后写入 timeline[0] 并将叙事追加到 narrative_full.txt。这是标准的 init 模式。

✅ 合理之处
  • 开场叙事 + 关系矩阵一次性生成，避免分步调用的不一致风险
  • tone 和 character_arc_outline 在 prompt 中可选注入，灵活自然
  • 关系值用 delta 而非绝对值应用：初始全50，LLM返回的值通过 trust - 50 转为增量
⚠️ 潜在问题
  • 名称匹配依赖精确字符串比较，LLM 稍有不规范即失效
  • 关系矩阵范围 -100~100，但碰撞阈值是 0~100 的逻辑，两者数值语义错位
名称模糊匹配缺失 中

_find_char_id_by_name() 使用 cfg.name == name 精确匹配。LLM 可能返回"林美雪"而配置中是"林 美雪"，或返回日文汉字对应关系错位。一旦匹配失败，关系数据静默丢失，无报错。

# story_engine.py — init_story() 中的名称匹配增强
import re

def _resolve_character_ref(self, name_str, configs):
  # 1. 精确匹配（最快）
  for cfg in configs:
    if cfg.name == name_str or cfg.name.replace(" ", "") == name_str:
      return cfg.character_id
  
  # 2. 移除空格和特殊字符后匹配
  clean_name = re.sub(r'[\s\u3000]', '', name_str)
  for cfg in configs:
    clean_cfg = re.sub(r'[\s\u3000]', '', cfg.name)
    if clean_name == clean_cfg:
      logger.warning(f"名称模糊匹配: '{name_str}' → '{cfg.name}'")
      return cfg.character_id
  
  # 3. 编辑距离兜底（需要 pip install python-Levenshtein）
  try:
    from Levenshtein import distance
    best = min(configs, key=lambda c: distance(clean_name, c.name))
    if distance(clean_name, best.name) <= 2: # 2字符以内差异
      logger.warning(
        f"编辑距离匹配: '{name_str}' → '{best.name}' "
        f"(距离={distance(clean_name, best.name)})")
      return best.character_id
  except ImportError:
    pass
  
  # 4. 全部失败 — 打 error 日志，不静默丢弃
  logger.error(f"无法匹配角色名称: '{name_str}'，可用角色: "
             f"{[c.name for c in configs]}")
  return None

关系数值语义与碰撞阈值不衔接 低

模块 S：日程初始化

S init_story_schedule()LLM 生成角色个人时间表 B ▶
逻辑评估

Per-agent schedule 是合理的调度策略，让每个角色自主决定时间，世界时钟"跳"到最早时间点。但初始化 prompt 中硬编码了 {base_hour + 2} 等示例值，这可能引导 LLM 偏好这些特定偏移量，削弱日程的自然多样性。

✅ 合理之处
  • 每个角色独立生成日程，尊重角色自主性
  • 跨日处理逻辑（hour ≤ base_hour → 次日）
  • LLM 失败时有 fallback 硬编码时间表兜底
⚠️ 潜在问题
  • prompt 示例值（+2/+4）可能引导 LLM 输出趋同
  • 跨日判定逻辑在边界情况有误判风险
  • 各角色日程独立生成，缺少"大家一起上课"的协调
跨日逻辑的边界歧义 中

代码中使用 actual_hour <= base_hour → 次日 判断跨日。但 LLM 被要求"小时必须 > 当前小时"。假设 base_hour=20（晚8点），LLM 返回 hour=22（晚10点，同一天），判断 22 > 20 → 当天；但 LLM 返回 hour=8（次日早8点），判断 8 ≤ 20 → 次日。这个逻辑依赖 LLM 正确理解"跨日"概念，而 LLM 可能不遵循，导致时间错乱。

💡 综合方案 — 参考 workflow-audit-v2 N7

方案 A（推荐）：使用总分钟数比较替代简单的小时比较

workflow-audit-v2 N7 指出当前 actual_hour <= base_hour → 次日 的判定在边界情况下有歧义。例如 base_hour=12、base_minute=30，LLM 返回 hour=12、minute=15——按小时比较 12≤12→判定为次日，但实际上 12:15 还在 12:30 之前，应该是今天更早的时间。

# story_engine.py — _resolve_schedule_day() 替代简单比较
def _resolve_schedule_day(self, raw_hour, base_hour, base_minute, current_day):
  target_total = raw_hour * 60 # 可扩展为 raw_hour*60 + raw_minute
  current_total = base_hour * 60 + base_minute
  if target_total <= current_total:
    return current_day + 1, raw_hour # 今天已过去 → 明天
  else:
    return current_day, raw_hour # 今天未来

将分钟也纳入比较，消除边界歧义。同时记录 warning 当检测到跨日事件以便调试。

方案 B（更彻底）：prompt 中要求 LLM 返回 relative_hour_offset

在 prompt 中显式要求 LLM 返回 relative_hour_offset（相对当前小时偏移，如 +2 表示 2 小时后），或直接使用 unix 风格的总分钟数。这样无需在 Python 端推断跨日，LLM 的意图更明确。但增加了 prompt 复杂度，且 LLM 计算偏移时可能仍有误差。

推荐方案 A——改动最小，逻辑最清晰。与 audit-v2 N7 方案一致。

群体日程缺少协调 低

如果这是一个校园剧，角色们理论上应该有共享课程、午休等时间段。目前每个角色独立生成日程，可能没有人同时出现在教室——这削弱了碰撞概率，减少了互动机会。

Step 1：世界时钟推进

1 _get_next_world_time() + Occupancy Model纯规则：最早时间点 + 占用管理 A ▶
逻辑评估

这是整个系统最稳健的模块。算法简单精确：取所有角色 schedule 的最小值，精确匹配的角色为"活跃"，弹出事件后推进。Occupancy Model 补充了"进行中活动不被中断"的约束。核心逻辑没有问题。

✅ 合理之处
  • 时间推进完全由角色日程驱动，无硬编码限制（符合用户铁律）
  • Occupancy Model 防止世界时钟打断正在进行的事件
  • 精确到分钟粒度，角色自主安排
  • schedule 为空时自动重新生成 + fallback
⚠️ 潜在问题
  • 精确匹配（day/hour/minute 三者全等）可能漏掉相近时间点的并发
  • 非活跃角色长时间无更新的"僵尸角色"问题
精确匹配导致"形离实合"的角色无法互动 中

如果角色A在 8:00 有事件，角色B在 8:05 有事件，两人同去图书馆——世界时钟先推进到 8:00，A 执行；再推进到 8:05，B 执行。两者不会被视为"活跃在同一时刻"，因此永远不会在图书馆相遇。这在 narrative 上可能不合理：5 分钟的差异在真实故事中近乎同时。

💡 代码级方案（原创，audit-v2 未覆盖）

方案：在 _get_next_world_time() 中引入时间窗口合并

# story_engine.py — 增强 _get_next_world_time()
TIME_WINDOW_MINUTES = 5 # 可配置

def _get_next_world_time(self):
  # Step 1: 找到最早时间点 t_min（保持现有逻辑）
  t_min = _find_earliest_time()
  if t_min is None: return None
  
  # Step 2: 时间窗口合并
  t_window_end = t_min + TIME_WINDOW_MINUTES # 以总分钟数表示
  active_cids = []
  for cid, state in character_states.items():
    for evt in state.personal_schedule[:1]: # 只看每个角色的下一个事件
      evt_time = _to_total_minutes(evt.day, evt.hour, evt.minute)
      if t_min <= evt_time <= t_window_end:
        active_cids.append(cid)
        break # 已纳入，不检查该角色后续事件
  
  # 但世界时钟保持跳转到 t_min（不跳跃）
  # 合并只影响"被视为活跃"的角色集合，不影响时间推进
  return t_min, active_cids

# 效果：A在8:00去图书馆，B在8:05去图书馆
# → t_min=8:00, t_window_end=8:05
# → active_cids = [A, B] → 两人都在图书馆，碰撞检测工作！
# 世界时钟仍跳转到 8:00（t_min），B 的 8:05 事件被提前弹出
# B 会在事件被消费后 8:00 的下一个事件是 schedule 中的下一条

注意事项：

  • 窗口内的角色被弹出时，其下一个 event 也被消费（pop），确保不会回头
  • Occupancy Model 仍然生效：如果角色在窗口内但被占用，不会被纳入活跃列表
  • 窗口大小建议 5 分钟，可通过 config.py 调整
  • 过大的窗口（>15 分钟）会导致"时间作弊"感——B 明明 15 分钟后才到，却被提前拉到一起
非活跃角色的"僵尸"状态 低

不在 active_char_ids 中的角色仅被赋予占位 intent（importance=2），不发生位置变化、心情变化。如果角色A的日程到 12:00 就结束了，而世界推进到 20:00（其他角色还在活动），A 会停留在最后的位置/心情/状态，连续多步没有任何更新。这在长期运行中可能产生不自然的"被遗忘"角色。

  avg_imp = sum(recent_imps) / len(recent_imps) if recent_imps else 4

# world_sim.py — 增强 _judge_encounter_type()
def _judge_encounter_type(rel_a, intent_a, intent_b, mood_a, mood_b):
  # Step 1: 基础阈值判定（保持现有逻辑）
  base_type = _base_threshold_judge(rel_a.trust)
  
  # Step 2: 情境修正因子（-0.3 ~ +0.3）
  modifier = 0.0
  
  # 因子 1: 情绪修正
  NEGATIVE_MOODS = {"愤怒", "沮丧", "焦虑", "悲伤", "烦躁"}
  if mood_a in NEGATIVE_MOODS or mood_b in NEGATIVE_MOODS:
    modifier -= 0.2 # 负面情绪 → 少说话
  
  # 因子 2: 意图紧急性修正
  URGENT_INTENTS = {"赶去", "追", "紧急", "迟到", "逃"}
  if any(kw in (intent_a + intent_b) for kw in URGENT_INTENTS):
    modifier -= 0.15 # 有急事 → 更少互动
  
  # 因子 3: 双方都空闲 → 更容易对话
  if _is_idle(intent_a) and _is_idle(intent_b):
    modifier += 0.15
  
  # Step 3: 修正后的阈值
  dialogue_threshold = 70 * (1 + modifier) # 修正后 49~91
  conflict_threshold = 30 * (1 - modifier) # 反向修正
  
  # Step 4: 最终判定

    return "DIALOGUE"
  elif rel_a.trust < conflict_threshold and rel_a.affection < 30:
    return "DIALOGUE" # 冲突
  elif abs(rel_a.trust - 50) < 15:
    return "OBSERVATION"
  else:
    return "PASSING"

# world_sim.py — _group_by_proximity() 替代简单 destination 分组
def _group_by_proximity(char_intents, char_states):
  groups = [] # 每组是一个 set of (cid, location)
  
  for cid, intent in char_intents.items():
    state = char_states[cid]
    dest = intent.get("destination") or state.current_location
    curr = state.current_location
    
    # 该角色可以出现在两个位置：当前 + 目标
    positions = {curr, dest} # 去重
    
    # 查找已有组
    merged = False
    for group in groups:
      # 检查该角色的任何位置是否与组内任何位置重叠
      group_locs = {loc for _, loc in group}
      if positions & group_locs: # 交集非空 → 相遇
        # 取交集位置作为遇地点（优先选 dest 如果都在组内）
        meet_loc = (positions & group_locs).pop()
        group.add((cid, meet_loc))
        merged = True
        break
    if not merged:
      groups.append({(cid, dest)}) # 新组，以目的地为准
  
  return groups

# 效果：角色A在教室→去走廊，角色B在走廊→去教室
# A: {教室, 走廊} B: {走廊, 教室} → 交集={教室, 走廊}
# → 两人在走廊相遇（优先取非自己当前位置的地点）

伏笔机制的根本性缺陷 高

Type B（伏笔留白）的执行逻辑是：在 check → normal 的低 importance 条件下，随机取出角色的 internal_thought 存入 foreshadow_buffer。然后在正常叙事时消费 buffer 中的内容注入 prompt。

问题在于：

  • internal_thought 是角色意图生成时的内心独白（如"今天午饭吃什么"），不是伏笔
  • buffer 的消费是 FIFO（取前5条），与当前叙事场景无关联
  • 过期伏笔直接写入记忆作为"[过期伏笔]"，对角色无实际影响

整个伏笔机制更像是"内心独白的延迟消费"，而非传统叙事意义上的伏笔（设置悬念 → 后续揭示）。

💡 综合方案 — 参考 workflow-audit-v2 N3

方案 A（立即修复，1行改动）：伏笔消费改为按 recency 倒序

workflow-audit-v2 N3 发现当前伏笔消费是 FIFO（取 buffer[:5]），导致最老的伏笔先被注入。如果 Step 8 产生的伏笔"走廊跟踪"比 Step 15 产生的"食堂菜单"更早入队，在 Step 18 食堂场景中，先注入的却是"走廊跟踪"——语境错位。

改为按 created_step 倒序消费：

# story_engine.py Line 569 改为：
take = sorted(self.state.foreshadow_buffer,
              key=lambda e: -e["created_step"])[:5]
# 消费最新的 5 条，而非最老的 5 条

好处：最新伏笔更可能与当前叙事语境相关，改动仅 1 行。

方案 B（中期重构）：由叙事 LLM 主动埋设伏笔

伏笔不应从 internal_thought（内心独白）提取，而应由叙事 LLM 在每次叙事时输出 "foreshadowing": "..." 字段——这是真正的"叙事性伏笔"（悬念、暗示、未完成的对话等）。在下次叙事 prompt 中注入这些未消费的伏笔。这与 workflow-audit P1-4 的伏笔过期机制（created_step + 8步过期）配合使用。

参考：audit-v2 N3 详细分析了 FIFO 导致的语境错位问题，方案 A 是成本最低的立即改进。方案 B 需要改动叙事 prompt，工作量更大但效果更好。

# story_engine.py — _build_story_summary() 增强
def _build_story_summary(self):
  all_narratives = self.state.timeline # 所有叙事步
  
  # Layer 1: 最近 20 步 — 保持完整（每条 200 字截断）
  recent = all_narratives[-20:]
  recent_text = "\n".join(n[:200] for n in recent)
  
  # Layer 2: 20-50 步前的 — LLM 压缩为精炼摘要
  older = all_narratives[:-20][-30:] # 最多取 30 条压缩
  if older and not self._cached_summary:
    compressed = await self._compress_narratives(older)
    self._cached_summary = compressed # 缓存，避免重复调用
  
  # Layer 3: 50 步之前的 — 使用已缓存的超压缩版本
  ancient = self._cached_summary or ""
  
  return f"## 近期发展\n{recent_text}\n\n## 前期概要\n{ancient}"

# 压缩函数 — 一次轻量 LLM 调用
async def _compress_narratives(self, narratives):
  prompt = """将以下故事发展压缩为 500-800 字的精炼摘要，
  保留关键情节节点、角色关系变化和重要事件。
  请用叙事性语言而非提纲。"
  {narratives}"""
  response = await llm_client.chat_async(prompt, max_tokens=500)
  return response

压缩触发时机：每新产生 10 条叙事时（而非每步），重新压缩一次。这样 10 步才消耗 1 次额外 LLM 调用，成本可控。

缓存失效：当故事存档/恢复时清除缓存，确保不同会话不共用压缩结果。

反思机制

逻辑评估

反思机制基于叙事步触发（P2-12 修复），取最近 6 步非反思记忆，生成洞察并更新 long_term_goal / daily_plan。整体合理。存在跨反思信息隔离的问题。

✅ 合理之处
  • 基于叙事步触发确保反思始终基于实际故事进展
  • 反思记忆 importance floor=7 + is_reflection 标记，检索时有加成
  • 可更新 long_term_goal 和 daily_plan，影响后续行为
⚠️ 潜在问题
  • 反思只看最近6步非反思记忆，忽略更早的上下文
  • 更新后的 goal/plan 在 prompt 中只是普通文本，LLM 可能忽略
反思缺乏长期记忆的连接 中

反思只基于最近6步的记忆，但"反思"这个行为本质上需要联系更久远的经历。角色在第 50 步反思时，只看第 44-50 步的记忆，看不到第 10 步发生过的重要事件。这限制了反思的深度。

💡 代码级方案（原创，audit-v2 未覆盖）

方案：双窗口记忆检索 — 近期窗口 + 远期采样

# story_engine.py — _run_reflections() 中的记忆检索
def _get_reflection_memories(self, cid, current_sim_step):
  all_memories = self.state.character_memories[cid]
  
  # 近期窗口：最近 6 步的非反思记忆
  recent_cutoff = current_sim_step - 6
  recent = [m for m in all_memories
          if m.get("step", 0) >= recent_cutoff
          and not m.get("is_reflection")]
  
  # 远期窗口：随机采样 2 条高 importance 旧记忆（step < recent_cutoff）
  distant = [m for m in all_memories
           if m.get("step", 0) < recent_cutoff
           and not m.get("is_reflection")
           and m.get("importance", 0) >= 5] # 只取重要的旧记忆
  if len(distant) > 2:
    distant = random.sample(distant, 2) # 随机 2 条
  
  # 合并：近期在前（权重更高），远期在后（提供深度）
  combined = recent + distant
  
  # 在 prompt 中区分展示
  context = "## 近期经历（最近几步）\n" + format(recent)
  context += "\n## 回顾过往\n" + format(distant)
  
  return combined, context

效果：角色在第 50 步反思时，既能看到第 44-50 步的近期"发生了什么"，也能随机回忆起第 10 步的"第一次见面"或第 25 步的"激烈争吵"——让反思真正有"回首往事"的深度。

跨模块结构性问题

🔴 问题1：关系系统完全静态

这是整个系统最根本的逻辑缺陷。关系矩阵在 init 时初始化（LLM 生成初始值），之后 trust/affection 数值永远不会变化。对话不更新关系，事件不更新关系，反思不更新关系。碰撞检测读取关系值来决定对话类型——但这些值是不变的，导致角色间的关系永远停留在初始状态。

影响："关系驱动叙事"的核心承诺无法兑现。A 和 B 经历了 50 步的故事后，关系数值仍然是初始值。碰撞判定看似动态实则静态。

解决方向（详见 Step 3 综合方案 + workflow-audit-v2 N4 思路）：

  • 对话 LLM 输出 relationship_delta：在 generate_dialogue prompt 中增加 {"trust": ±N, "affection": ±N} 输出，Step 4 中调用 relationship.update_relationship() 应用
  • OBSERVATION 碰撞产生微量关系变化：旁观者对对话参与者产生 ±1 trust 偏移
  • 反思触发关系重新评估：反思 prompt 输出 reevaluated_relationships 字段
  • 世界事件自动调整受影响角色的关系：在 _judge_event_impact 中增加关系影响输出
  • ⚠️ 前置依赖：修复 audit-v2 N4 对话异常静默吞没问题，确保 dialogue 字段始终可用

🟡 问题2：LLM 输出不确定性与下游逻辑的脆弱耦合

系统依赖 LLM 返回结构化 JSON，但 LLM 偶尔会返回不符合预期的值（null、错误的 location 名称、不合理的时间偏移）。虽然有 null 防御和 anti-reverse 校验，但这些都是"底线防御"而非"主动保证"。

具体脆弱点与对应方案：

  • destination 名称不规范：LLM 可能返回非标准名称 → 使用标准化地点列表约束 prompt 输出（audit P0-4 方案 A）
  • next_event 无上限校验：LLM 可能返回 72 小时后 → 增加最大偏移限制 +8h（本报告 Step 2 方案）
  • mood 无标准化：LLM 可能返回非标准表达 → 在 prompt 中约束为预设情绪列表
  • JSON 格式遵守率不稳定：所有 prompt 仅含 system role → 追加 user message trigger（audit-v2 N6，改动覆盖 6 个 prompt 模块）
  • 并发调用无控制：22 个并行 LLM 调用可能触发限流 → 引入 asyncio.Semaphore(6)（audit-v2 N8）
  • importance 评分不一致：跨步缺乏一致性 → 在 prompt 中给出评分锚点参考

🟡 问题3：角色意图与叙事之间的割裂

Step 2 生成 intent（"去图书馆借书"），Step 3 生成对话（"在图书馆遇到了XX"），Step 5 生成叙事。但叙事 LLM 可能会重新解释或改写角色行为——它看到的是一份 interaction 记录，而非原始 intent。

例如：角色A的 intent 是"独自去天台冷静一下"，importance=6。叙事 LLM 看到这个 interaction 后可能写成"A 落寞地走向天台"——但也可能写成"A 气冲冲地去天台找谁算账"。叙事 LLM 有创作自由度，但这种自由度可能导致角色行为被扭曲。

风险：如果意图和叙事之间有冲突，下一步的 memory 存储的是叙事内容（而非原始意图），角色记忆可能被"污染"。

解决方向：

  • 在叙事 prompt 中将 intent 作为"行为约束"注入："以下角色的行为意图是确定的，请勿在叙事中偏离其核心行动"
  • 叙事输出增加 intent_fidelity 自评字段（LLM 评估叙事是否偏离了意图）
  • audit P1-1 的统一单次叙事调用方案也有助于此：LLM 看到全局时间线后更不容易偏离单个角色的行为

🟢 问题4：物理体感缺失

系统没有"饥饿/疲劳/时间流逝"的物理体感机制。角色可以永远不吃饭、不睡觉（虽然日程可能安排这些）。physical_condition 字段由 LLM 通过 condition_change 自行管理，没有系统级的强制衰减。这意味着角色的身体状态完全依赖 LLM 的"自觉"，缺少系统兜底。

问题不大但值得注意：在长时间运行后，physical_condition 可能偏离合理范围（如连续 50 步都是"精力充沛"）。

解决方向：

  • 每步自动对 physical_condition 做轻微衰减（如每 10 步将"精力充沛"降级为"略有疲惫"）
  • 当角色 schedule 中包含"用餐"类型事件时，自动恢复身体状态
  • 将 physical_condition 与重要性联动：疲惫时 importance 评分自动降低
