问题 1：照桥心美 Step 5 人设严重不符

1 照桥心美 Step 5 — 人设被 LLM 自行修改 叙事质量 ▶
人设定
background: "从出生就被所有人宠爱，无论走到哪里都是焦点，从未尝过被拒绝的滋味"
personality: "表面完美无缺、温柔优雅，内心极度自信甚至自负"
fear: "在众人面前出丑，或者失去'完美'的标签"
motivation: "让全世界（尤其是齐木楠雄）都为自己着迷"
speaking_style: "笑容满分，但内心会疯狂OS'哦呼！''怎么还不哦呼！'"
secret: "对齐木楠雄产生了前所未有的执念，因为他是唯一一个不对自己'哦呼'的人"
Step 5 实际产出（与设定矛盾的部分）
"教室里的男生们要么在讨论游戏，要么在打瞌睡，偶尔有人抬头看了她一眼——然后，就只是看了一眼"

"照桥心美的笑容僵在脸上，内心的小剧场开始疯狂上演：'怎么回事？！为什么没人"哦呼"！'"

"那个男生挠了挠头，开口道：'照桥同学，你的英语课本…拿反了。'说完，他头也不回地走了。"

"她的手指不自觉地攥紧了课本边缘——力道大得书页都有了褶皱"

🔗 流程根因追溯

Step 5 叙事编排 → LLM 收到 prompt → prompt 含角色卡 → 但 LLM 优先级：写"有趣叙事" > 遵守人设

根因：叙事 prompt 没有"人设一致性"硬约束

角色卡中包含完整人设（background/personality/fear 等），但 LLM 在创作叙事时优先考虑"戏剧张力"——让完美女神遭遇小尴尬是经典叙事套路——而忽略了这与人设的根本矛盾。当前 prompt 没有类似 "叙事必须严格遵守角色设定，不得让角色出现与其背景矛盾的行为" 的硬约束。

更深层的问题：系统中没有"人设校验层"。Step 2 生成 intent 时角色人设通过 character_action prompt 约束行为（效果较好），但 Step 5 叙事时 LLM 的创作自由度较大，可以"改写"角色的行为表现。intent 只是"在走廊漫步展现姿态"，LLM 把这个改写成了"遭遇尴尬"的场景。

流程级修复

在 narrative prompt 中增加人设一致性约束："你正在为角色 {name} 写叙事。以下是她的性格设定，叙事中的所有行为必须与设定一致。任何人设禁止的行为不得出现：{background} + {fear} + {secret}"

中期方案：在叙事 prompt 中注入角色的 "key constraints"（如"照桥心美：绝对不能出现被忽视/被拒绝的场景；海藤瞬：中二台词后必须立刻心虚；齐木楠雄：永远不会主动开口说话"）。

🔗 流程根因追溯

弧追踪 cooldown → Step 1/3 都是 full 叙事 → story_summary 含 Step 1 叙事 → LLM "延续"了 Step 1 的风格和开头 → 两个 routine 事件 → 雷同叙事

根因：相似上下文 + 无"去重"指导 = LLM 惯性重复

直接原因：Step 1 和 Step 3 都在"教学楼楼梯拐角"，角色都是海藤瞬，intent 都涉及"观察/隐藏"。story_summary 包含 Step 1 的完整叙事，LLM 自然"延续"了相同的描写方式。

根本原因：如果 skip 正常工作，Step 3 的 importance=3 应该被跳过——根本不会生成叙事。但因为 cooldown 强制 full 叙事（见 skip-root-cause.html），两个相似场景都生成了完整的叙事。同时 narrative prompt 没有"如果当前场景与上一步高度相似，请不要重复相同描写"的指导。

流程级修复

短期：修复 skip（见 skip-root-cause.html）→ Step 3 直接被跳过，问题自然消失。

中期：narrative prompt 增加去重约束："如果这一步的场景和角色与上一步完全相同，请聚焦于与前一步不同的新细节，不要重复相同的环境描写和开篇句式。"

🔗 流程根因追溯

LLM 日程初始化 → 生成"09:00 上课画符号" → _pop_scheduled_events 取出 → Occupancy 创建：locked_location = current_location → current_location = "教学楼楼梯拐角"（initial） → 占位意图 destination = locked_location = "楼梯拐角" → "上课" + "楼梯拐角" 矛盾！

根因：Occupancy 的 locked_location 基于当时的 current_location 而非 activity 的合理位置

Occupancy Model 在创建占用时使用 current_location 作为 locked_location（源码 line 232）。海藤的初始位置是"教学楼楼梯拐角"，Step 1 他在这里活动，Step 3 他的下一个 schedule item 开始时，他"还在这里"——因为系统没有让他在 Step 1 和 Step 3 之间移动到"教室"。

深层问题：schedule 中的事件按时间先后排列，但事件之间的"过渡"（海藤从楼梯拐角走到教室）没有被建模。LLM 在生成 schedule 时知道"09:00 是上课"，但 Occupancy Model 不知道——它只看 current_location。角色位置在事件边界上不自动更新到合理值。

流程级修复

方案A（轻量）：在 Occupancy 创建时，如果 intent description 包含地点关键词（如"上课"→"教室"、"吃拉面"→"食堂"），用规则映射覆盖 locked_location，而不是无条件使用 current_location。

方案B（中等）：在 Step 2 intent 或 schedule init 中让 LLM 返回 location 字段（而非只用 destination），使角色位置随活动自动更新。

方案C（结构化）：为每个 Location 定义"典型活动"映射（教室→上课/自习，食堂→用餐，图书馆→自习/看书），Occupancy 创建时自动匹配。

🔗 三个流程缺陷叠加导致零互动

缺陷①：日程时间错开
Module S: 各角色独立生成 schedule → 海藤 08:15, 照桥 08:30, 齐木 12:00, 燃堂 10:30
没有共享时间段，一个角色活动时另一个在"等待中"
缺陷②：非活跃角色不参与碰撞
Step 1: _get_active_characters 精确匹配 → 仅活跃角色进入 collision group → 齐木和燃堂在教室但未被分组
即使物理同处一室，非活跃角色被碰撞检测排除在外
缺陷③：Occupancy 锁定位置
角色被占用 → location 锁死 → 海藤永远在楼梯拐角，照桥永远在教室过道
即使 schedule 中有"午休""课后"等不同场景，角色也不会移动

根因总结

三个缺陷的根源各不相同，但它们联手制造了"零互动"的结果。最关键的修复是缺陷②：碰撞检测应该考虑所有同位置的角色（包括非活跃角色），而不是仅限活跃角色。非活跃角色虽然不在"做"任何事，但他们在场——在场就是产生互动的条件。

流程级修复优先级

P0：碰撞检测纳入非活跃角色。修改 detect_encounters()：对同 location 的所有角色对做碰撞检测（不仅限活跃），非活跃角色自动生成一个 passive intent（"在座位上坐着"）参与碰撞。

P1：日程共享时间段。Module S 在初始化时先由系统生成共享框架（上课/午休/社团），角色在框架内个性化。提高同时间同地点的概率。

P2：时间窗口合并。±5 分钟的相近时间点一起激活，让 08:15 和 08:30 的角色同时在场。

问题 5：Skip 完全失效

5 Skip 失效 — 6步全部生成叙事 系统机制 ▶

详细分析见 skip-root-cause.html。根因：LLM 日程 importance ≥ 7 → Occupancy 透传 → 占位意图 max_importance ≥ ARC_HIGH_IMPORTANCE → cooldown → skip 不可能。修复：line 326 加 min(imp, 6) 截断。

问题 8：时间倒退

8 时间倒退 — Step 6 @ 09:16 在 Step 5 @ 10:00 之后 系统机制 ▶

详细分析见 skip-root-cause.html 第五部分。根因：N2 自动事件写入 busy_until+1 时间，当角色"错过激活窗口"时，写入的时间已被世界时钟超越。修复：N2 写入时间 = max(busy_until+1, 当前世界时钟)。

🔗 因果关系链

日程时间错开 → 每次仅1个角色活跃 → 同位置无其他角色 → 碰撞检测返回空 → 零 dialogue → 零 relationship_delta → 关系矩阵冻结

根因：这是问题 6（零互动）的必然结果

关系 updating 的入口只有一个：dialogue 产生的 relationship_delta。如果永远没有 dialogue，关系就是静态的。P0 关系闭环修复添加了 relationship_delta 机制，但没有为"非对话场景"提供关系变化入口——比如 OBSERVATION（旁观）应该微量改变关系，反思应该触发关系重新评估。

齐木和燃堂从未激活的原因更直接：他们的 schedule 起始时间晚（齐木 12:00、燃堂 10:30），而模拟只跑了 6 步（到 10:16 结束）。这是模拟长度不足 + early-schedule characters 的"首步偏差"。

流程级修复

关系不变：修复问题 6 后自然改善。另外增加 OBSERVATION → 微量关系变化（+1 trust）、反思 → 关系重新评估。

齐木燃堂未登场：非 bug，模拟步数少 + 他们 schedule 晚是正常现象。如果希望首步就有全员参与，可以在 schedule 初始化时为所有角色强制加一个"08:00 到校"的共享事件。
