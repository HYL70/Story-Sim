🔗 从"LLM 日程生成"到"cooldown 锁定"的 6 层传递链

第1层：LLM 日程生成
_initialize_personal_schedules() → LLM 为每个角色生成 3-5 个日程，自带 importance 字段 (1-10)
源码：story_engine.py:798 prompt 示例值=5，但 LLM 实际可输出 1-10 任意值
第2层：存入 PersonalEvent
PersonalEvent(importance=imp) → 存入 state.personal_schedule
源码：story_engine.py:851 — imp = item.get("importance", 5)
第3层：弹出事件 → 创建 Occupancy
_pop_scheduled_events() → scheduled_events[cid] = { ..., "importance": evt.importance }
→ occupancy[cid] = {"importance": evt.get("importance", 5)}
源码：story_engine.py:234 — 原始 importance 原封不动透传
第4层：占位意图继承 importance
Occupancy 占位 → intents[cid] = {"importance": occ.get("importance", 5)}

第5层：max_importance 计算
build_interactions() → max_importance = max(ia.importance for ia in active_interactions)
源码：story_engine.py:449 — 占位意图的 importance 作为正常值参与 max
第6层：cooldown 触发 → skip 不可能

三、为什么 LLM 会把低重要性活动赋高分？

日程初始化 prompt 分析

# story_engine.py:796-808 — 日程生成 prompt {{"schedule": [ {{"hour": {base_hour + 2}, "minute": 30, "duration": 60, "description": "what to do", "importance": 5}}, {{"hour": {base_hour + 4}, "minute": 0, "duration": 90, "description": "what to do", "importance": 4}} ]}} ... - importance: 1-10

问题：prompt 中 importance 的示例值是 5 和 4，但提示文字只说 "1-10"——没有任何指导规则告诉 LLM 如何区分"routine"和"significant"。LLM 从角色设定推断："照桥心美认为自己的每一个举动都很重要" → 给所有活动赋高分。这是合理的 LLM 推理，但不符合系统的设计意图：routine 活动应该是 importance 3-5，只有真正的情节转折才应在 7+。

四、为什么这个问题特别严重？

cooldown 路径上的迭代循环

一旦一个角色的 routine 活动触发 cooldown → 连续 3 步全部 full 叙事。3 步后进入 check → 回到 normal → 但下一个活跃角色可能又是一个 routine 活动 → 再次触发 cooldown → 又 3 步 full 叙事。

结果：系统大约 75% 的步骤都在 cooldown 状态中（3 cooldown → 1 check → 3 cooldown ...）。normal 状态只在 check→normal 的那一瞬间存在，几乎立即被下一次 cooldown 覆盖。

Step1 08:15 [cooldown →] 高压事件！进入 cooldown ← LLM schedule imp≥7 Step2 08:30 [cooldown ×] cooldown 期间，完整叙事 Step3 09:00 [cooldown ×] cooldown 期间，完整叙事 Step4 09:01 [check → normal] Type A 轻量叙事 ← N2 事件 imp=3 Step5 10:00 [normal → cooldown] 高压事件！进入 cooldown ← LLM schedule imp≥7 Step6 09:16 [cooldown ×] cooldown 期间，完整叙事 ⚠️ 时间倒退！

五、附：时间倒退问题（Step 6 @ 09:16 在 Step 5 @ 10:00 之后）

N2 自动事件写入"过去时间"导致时间倒退

Steps 2-4 中 照桥心美 处于占用状态 (08:30 → 09:15)，期间未激活。到 Step 5 世界时钟推进到 10:00 时，照桥心美的占用清理才触发：

# story_engine.py:204-214 — overhead delay 效应 for cid in expired: busy_until = occ.get("busy_until") # = (1, 9, 15) state.personal_schedule.append(PersonalEvent( day=busy_until[0], # = 1 hour=busy_until[1], # = 9 minute=min(busy_until[2] + 1, 59), # = 16 ← 09:16 已被世界时钟超越！ ... ))

问题：N2 事件写入 (1, 9, 16)，但世界时钟已是 (1, 10, 0)。下一步 _get_next_world_time 找到最小时间 (1,9,16) → 时间倒退 44 分钟。这导致 照桥心美 在"过去时间"再做一次"自由活动"，而实际已经发生了 10:00 的课间漫步。

正确做法：N2 写入时应 max(busy_until 后 1 分钟, 当前世界时钟)，防止写入过去时间。

六、修复方案

主修复：截断 Occupancy 占位意图的 importance 推荐

位置：story_engine.py:326

原理：Occupancy 占位意图代表"角色正在做预定的事"，不是"角色主动做出的戏剧性决定"。routine 活动不应该触发 cooldown。

# 当前代码 (line 326) imp = occ.get("importance", 5) # 修复方案A（最小改动）：cap 到 6，确保不触发 cooldown imp = min(occ.get("importance", 5), 6) # 修复方案B（语义更正确）：默认给占位意图固定低值 imp = 3 # routine 活动的固有重要性

推荐方案A：cap 到 6 比直接固定 3 更温和——保留了"某些 scheduled event 确实比其他 routine 更重要"的信息（imp=4 vs imp=6 仍有差异），但确保不会跨越 ARC_HIGH_IMPORTANCE=7 的 cooldown 阈值。

副修复1：日程生成 prompt 明确 importance 语义

位置：story_engine.py:807

当前：importance: 1-10

改为：

importance: 1-10 (1-3=trivial, 4-6=routine, 7-9=significant event, 10=life-changing) Most daily activities should be 3-5. Only give 7+ if this activity represents a major plot point or emotional turning point.

副修复2：N2 时间倒退防御

位置：story_engine.py:206-209

当前：N2 写入 busy_until + 1 minute（可能已过去）

# 修复：N2 写入时间不小于当前世界时钟 n2_minute = busy_until[2] + 1 n2_hour = busy_until[1] + n2_minute // 60 n2_day = busy_until[0] + n2_hour // 24 n2_minute = n2_minute % 60 n2_hour = n2_hour % 24 # 确保不早于当前世界时钟 n2_total = n2_day * 1440 + n2_hour * 60 + n2_minute current_total = day * 1440 + hour * 60 + minute + 1 if n2_total < current_total: n2_day, n2_hour, n2_minute = day, hour, minute + 1 if n2_minute >= 60: n2_hour += 1; n2_minute = 0

七、总结

结论

Skip 完全不生效的根本原因：LLM 在日程初始化时给 routine 活动赋予了 importance ≥ 7。这个值通过 6 层传递链（PersonalEvent → _pop → occupancy → 占位意图 → max_importance → arc_state）最终触发 cooldown。cooldown 状态下 skip 不可能发生。

核心修复（line 326，1 行改动）：imp = min(occ.get("importance", 5), 6) —— 截断到 6，确保 routine 活动不越过 ARC_HIGH_IMPORTANCE=7 的阈值。 配合日程 prompt 中的 importance 语义说明，可以从源头减少 LLM 赋高分的概率。

时间倒退是独立问题：N2 自动事件在 occupancy 清理时机与角色"错过激活窗口"叠加时，会写入过去时间。修复方案与原修复逻辑不冲突，可独立处理。
