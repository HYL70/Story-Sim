二、修复质量评估

📝
N1 修复附加分析：cooldown→check 过渡行为发生变化
story_engine.py:430-480
新观察 行为变化
▶
变更分析

N1 修复前：arc_state（局部快照）= "cooldown" → 无论 self.state.arc_state 是否已变为 "check"，都进入 full 叙事。

N1 修复后：统一使用 self.state.arc_state → 当 cooldown_remaining 减到 0 后 state 切换为 "check"，叙事判定也走 check 分支。

实际影响：cooldown 的最后一步从"始终完整叙事"变为"check 判定"。check 分支会根据 importance 和碰撞情况选择 full/light/foreshadow 三种模式之一：

  1. 有碰撞或有意义事件(imp≥4) → full 叙事（与原行为一致）
  2. imp≥3 且无碰撞 → light 叙事（比原行为更轻）
  3. imp<3 且无碰撞 → foreshadow 伏笔留白（比原行为轻得多）

评估：这不是一个 bug，而是一个合理的改进——原行为中 cooldown 最后一步"强制 full"可能导致一段无内容的冗余叙事。现在的 check 逻辑更精确地判断是否值得叙事。但需注意：如果剧情弧在高潮后迅速平静（cooldown 期间无新事件），cooldown 的最后一步可能被跳过（foreshadow 模式），读者感知到的高潮余韵期被缩短。

新发现：反思触发时所有 active 角色均参与，但只应在"有叙事"时被触发
story_engine.py:667-669 · _run_reflections()
新发现 反思频率 性能
▶
源码证据
# Lines 667-669: 基于 narrative_step_count 触发反思（全量 active 角色） should_reflect = ( self.state.narrative_step_count > 0 and self.state.narrative_step_count % config.REFLECTION_INTERVAL == 0 ) if should_reflect: await self._run_reflections(active_char_ids, step, time_label) # ← 所有活跃角色 # Lines 1189-1222: _run_reflections — 对每个 cid 调 LLM + 解析 for cid in char_ids: recent_mem = memory.get_recent_memories(...) if not recent_mem: continue # 无记忆则跳过（轻微优化） # LLM 调用 / JSON 解析 / 写入记忆...
问题分析

当前 _run_reflections 遍历 所有 active_char_ids（活跃角色），而非仅限活跃的交互动角色或所有角色。这导致：

  1. 过度触发：一节课只有 2 个角色在上课（活跃于当前时间），反思也只在这 2 个角色上运行。其他 6 个"不活跃"的角色永远不会被反思。但如果当前时间有 8 个角色都活跃（课间休息），则所有 8 个角色同时触发反思 = 8 次 LLM 调用。
  2. 覆盖率不均：活跃频率低的角色（如只在特定时间出现的配角）反思次数极低，可能 20 步才有一次反思。
  3. 与 Stanford 论文差异：原论文在设计上，反思是每天结束时的全量操作（所有角色在夜间反思）。当前方案是每 3 个叙事步仅对活跃角色反思，活跃度直接决定了反思频率。

严重程度：中等 — 不活跃角色的认知发展受限（长期后果），但短期内不影响故事连贯性。主要是设计取舍问题。

推荐方案
方案 A：反思覆盖所有角色（而非仅活跃角色） 每隔 REFLECTION_INTERVAL 个叙事步，对所有角色（而非仅 active）运行反思。不活跃角色的反思也可以基于其最近记忆生成。 if should_reflect: await self._run_reflections( list(self.state.character_states.keys()), # 所有角色 step, time_label ) 方案 B：按天触发反思（Stanford 方案） 在跨日时（current_day 变化）触发所有角色的反思，而非基于叙事步计数。更自然——角色在每天结束时反思。 推荐 A（改动最小，1 行变更），后续可考虑 B。

代码中没有任何 asyncio.Semaphore 或类似的并发控制。在每个时间步中：

问题分析

旧路径将玩家事件写入所有角色的 personal_schedule，破坏了 per-agent schedule 的核心理念。且：

  • 写入 day=self.state.current_day — 如果当前时间是 16:00，设定 hour=8 意味着事件在过去（可能触发时间倒退保护）
  • importance=8 硬编码 — 所有玩家事件同等重要
  • 不经过 _judge_event_impact() — 事件不会判定影响哪些角色
推荐方案
统一走新路径：
  1. 检查 Gradio UI 中 add_player_event 的调用方是否仍传 hour 参数
  2. 如果不传 → 删除旧路径分支
  3. 如果仍传 → 修改 UI 去掉 hour 参数，或在 WorldEvent 模型中增加 scheduled_hour 字段
  4. 将所有玩家事件统一走 world_event.add_player_event → pending_player_events → _judge_event_impact 路径
