Step 1：世界时钟推进

N2 修复：占用角色在 occupancy 过期清理时自动写入 PersonalEvent("自由活动", importance=3, source="auto")。事件设置在 busy_until 时间点的下一分钟，角色不会从调度系统中消失。

Node E：世界事件处理

E process_pending_events() + auto_event B（↑ B→B+） ▶
P0 消费修复：在 Step 2 之前构建 char_affected_events 索引（遍历 active_events 的 affected_characters 字段），intent 生成时传入 affected_events 参数。事件影响判定结果从"写入即忘"变为"被下游消费"。

⚠️ 仍存在的问题：玩家事件 importance 固定为 8

所有玩家事件 importance=8（始终触发 cooldown）。这是设计选择而非 bug——玩家干预意味着"有大事发生"。但如果玩家快速连续添加多个事件，会连续触发 cooldown，节奏受影响。

这修复了什么？

在第一轮报告中，Step 3 被评为 C 级（最差），主要因为"关系数值只读不更新"——这是整个系统最严重的设计缺陷。现在 relationship_delta 机制建立了关系更新的闭环：

  • 对话 LLM 输出 trust/affection 变化量（由对话内容驱动）
  • Story 在 Step 3 处理对话后立即应用增量
  • 下次碰撞检测时读到的是更新后的关系值 → 对话类型可能改变
  • 关系矩阵从静态变成了动态的

🔴 新发现：dialogue_data 的存储方式有问题

world_sim.py 中使用 object.__setattr__(encounter, 'dialogue_data', result) 将完整 LLM 结果附加到 Encounter 对象。但 Encounter 是 Pydantic BaseModel，这种方式绕过了 Pydantic 的验证机制。更严重的是，story_engine.py 中使用 hasattr(enc, 'dialogue_data') 来检测——如果一个 Encounter 是 Pydantic 反序列化出来的（从存档加载），它不会有这个动态属性，关系更新会静默失败。

建议：在 Encounter 模型中增加一个 dialogue_data: Optional[dict] = None 字段，或直接增加 trust_delta: int = 0 和 affection_delta: int = 0 字段。

⚠️ 仍存在：碰撞硬阈值忽略情境

_judge_encounter_type 仍然是纯阈值判定（trust > 70 → 对话），没有 mood/urgency 修正因子。

修复引入的 2 个新问题

🔴 新问题 1：dialogue_data 使用 object.__setattr__ 绕过 Pydantic

world_sim.py 第 142 行：

object.__setattr__(encounter, 'dialogue_data', result)

问题：Encounter 是 Pydantic BaseModel 实例。通过 object.__setattr__ 设置属性绕过了 Pydantic 的类型验证、序列化和模型导出。在 story_engine.py 中通过 hasattr(enc, 'dialogue_data') 检测——如果 Encounter 是从存档反序列化出来的，这个属性不存在，关系更新会静默失败。

风险：存档/读档后关系数值不更新。

建议修复：在 Encounter 模型中增加 dialogue_data: Optional[dict] = Field(default=None, exclude=True) 字段。

🟡 新问题 2：relationship_delta 的兜底路径为空

story_engine.py Step 3 处理 dialogue 后的关系更新代码：

if not delta and hasattr(enc, 'outcome'):
    # 兜底：LLM 可能未返回 relationship_delta，尝试从对话结果推断
    continue # ← 什么都没做！

问题：注释说"尝试从对话结果推断"，但代码实际是 continue——跳过该 encounter，不做任何关系更新。这意味着如果 LLM 在某次对话中没输出 relationship_delta，这次对话对关系的影响被完全丢失。

建议：要么实现启发式推断（从对话的 outcome 和 dialogue 内容中猜测情感方向），要么把 relationship_delta 设为 prompt 中的 required 字段（而非 optional），在 world_sim.py 解析时加 assertion。

总结

本轮修复的核心成就是三个 P0 结构性缺陷的消除：

  • 关系闭环：对话 LLM 输出 relationship_delta → world_sim 保存 → story_engine 应用 → 下次碰撞检测读到新值。这是从"静态关系"到"动态关系"的质变。
  • 事件差异化：affected_characters 从"写入即忘"变为下游消费，角色根据自己是否受影响做出不同反应。
  • 情境感知记忆：检索从纯时间/重要性变为 multi-factor（地点+心情+关系+计划），使记忆检索有了相关性维度。

7 个代码级 bug（N1-N7）全部修复，其中 N2（占用角色消失）和 N5（重复事件）是最危险的 bug，N4（静默异常）是最隐蔽的。反倒退校验（ANTI_REVERSE_MIN_MINUTES）为 LLM 输出的时间异常提供了系统级兜底。

剩余 6 个问题按优先级分为两类：

  • 立即可修（低风险、小改动）：N8 并发控制（加 1 个 Semaphore）、N9 双路径废弃（确认 UI 后删旧代码）、新问题 1 dialogue_data（加 1 个模型字段）
  • 设计迭代（需要测试验证）：碰撞情境修饰、时间窗口、light_mode 差异化、auto_event 动态间隔
